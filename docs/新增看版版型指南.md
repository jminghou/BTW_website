# 數位看版 — 新增版型指南

> 本文件說明：當你要為電子看版新增一款**新的樣式版型**時，需要準備哪些檔案、放在哪個路徑、以及正規的操作流程。
> 適用對象：美編／維護人員。最後更新：2026-05-19。

---

## 一、先搞懂三個概念

### 1. 「版型」到底是什麼？

一款版型 = **一組 CSS + JS + 圖片/字型資源** + **引用它們的菜單 HTML**。

例如目前的 `vis_ad_machine` 版型由這些檔案組成：
- `css/vis_ad_machine.css`（樣式）
- `js/spotlight_slideshow.js`（輪播邏輯）
- `pic/`（logo、編號圖示、背景圖等）
- `font/`（字型）

菜單 HTML 檔案本身（例如 `F3_D_2026-05-02.html`）會在開頭引用這組資源。

### 2. 兩種檔案，兩種存放方式（重要！）

| 類型 | 範例 | 存放位置 | 怎麼上線 |
|---|---|---|---|
| **版型資源**（共用、少變動） | css / js / pic / font | 程式專案的 `public/signage-assets/` | **git push 重新部署** |
| **菜單 HTML**（每天/每週變動） | `F3_D_2026-05-02.html` | Vercel Blob（雲端） | **後台「素材」直接上傳，免部署** |

> 🔑 **關鍵差異**：
> - 加「新版型」要動到 `public/signage-assets/`，**必須 git push 重新部署**才會生效。
> - 日常上傳「菜單」只是在後台點上傳，**不需要部署**。

### 3. 路徑是怎麼運作的？

菜單 HTML 用「相對路徑」引用資源，系統的 proxy 會自動改寫成正確的網址：

```
你的 HTML 寫：     ../../css/你的版型.css
系統自動改寫成：    /signage-assets/css/你的版型.css
實際讀取檔案：      public/signage-assets/css/你的版型.css
```

CSS 內部若有 `url('../font/xxx.ttf')`，瀏覽器會自動對應到 `public/signage-assets/font/xxx.ttf`，**不用你操心**。

## 二、資料夾路徑總覽

所有版型資源都放在這個根目錄下（**共用資源池**，所有版型混放，靠檔名區分）：

```
BTW_website/public/signage-assets/
├── css/      ← 各版型的 .css（例：vis_ad_machine.css、vis5_style.css）
├── js/       ← 各版型的 .js（例：spotlight_slideshow.js）
├── pic/      ← 圖片（logo、編號、背景…）
│   ├── logos/
│   ├── numbers/
│   └── qrcode/
└── font/     ← 字型 .ttf / .woff2
```

⚠️ **資料夾名稱只能是 `css`、`js`、`pic`、`font` 這四個**（見第五節說明）。

## 三、新增版型的正規流程（7 步驟）

### Step 1 — 準備版型資源檔

把新版型需要的檔案準備好：
- 1 個 CSS 檔，命名要**獨一無二**，例如 `modern_2026.css`
- 若有專屬 JS，命名也要獨一無二，例如 `modern_2026.js`
- 新增的圖片、字型

### Step 2 — 放進對應資料夾

| 檔案 | 放到 |
|---|---|
| `modern_2026.css` | `BTW_website/public/signage-assets/css/` |
| `modern_2026.js` | `BTW_website/public/signage-assets/js/` |
| 新圖片 | `BTW_website/public/signage-assets/pic/`（或其子資料夾） |
| 新字型 | `BTW_website/public/signage-assets/font/` |

### Step 3 — 確認菜單 HTML 的引用路徑正確

新版型的菜單 HTML（由匯出工具產生）開頭必須這樣引用，**用 `../../` 兩層相對路徑**：

```html
<!-- ✅ 正確：兩層相對路徑，folder 名稱是 css/js/pic/font -->
<link rel="stylesheet" href="../../css/modern_2026.css">
<img src="../../pic/VIS_logo.png">
<img src="../../pic/numbers/01.png">
<script src="../../js/modern_2026.js"></script>

<!-- 餐點圖片用完整網址，不受影響 -->
<img src="https://www.bythewaylife.com.tw/btwFile/Meals/xxx.png">
```

> CSS 檔案內部則用**一層** `../`：
> ```css
> @font-face { src: url('../font/你的字型.ttf'); }
> .bg { background: url('../pic/你的背景.png'); }
> ```

### Step 4 — 提交並部署（讓資源上線）

因為資源放在 `public/`，必須重新部署：

```bash
cd BTW_website
git add public/signage-assets
git commit -m "feat(signage): 新增 modern_2026 版型資源"
git push
```

git push 後 Vercel 會自動部署（約 1–2 分鐘）。部署完成後，資源網址才會生效。

> **驗證資源已上線**：瀏覽器直接開
> `https://你的網域/signage-assets/css/modern_2026.css`
> 看得到 CSS 內容 = 成功。

### Step 5 — 上傳菜單 HTML（後台操作，免部署）

1. 進入 **後台 → 電子看版 → 素材**
2. 選擇所屬廠區
3. 上傳該版型產生的菜單 .html（可一次多檔）

> 💡 若你有的是**餐點資料 JSON**（而非現成 HTML），可改用「上傳 JSON 自動轉檔」模式，
> 系統會自動拆分並產生菜單 — 詳見 [菜單 JSON 上傳轉檔指南](菜單JSON上傳轉檔指南.md)。

### Step 6 — 建立播放清單

1. **播放清單** 分頁 → 新增清單
2. 點「編輯項目」→ 把剛上傳的菜單加入 → 設定每張秒數 → **儲存項目**

### Step 7 — 設定排程

1. **排程** 分頁 → 新增排程（或批次套用到多螢幕）
2. 指定螢幕、播放清單、時段、星期（或特定日期）

完成後，螢幕端開 `/signage/player/[螢幕 key]` 即會播放新版型菜單。

## 四、快速檢查清單

新增版型前後，逐項確認：

- [ ] CSS 檔名獨一無二（不會跟現有版型撞名）
- [ ] 資源放對位置（`public/signage-assets/{css,js,pic,font}/`）
- [ ] 資料夾名稱只用 `css` / `js` / `pic` / `font`
- [ ] 菜單 HTML 用 `../../` 兩層相對路徑引用 css/js/pic
- [ ] CSS 內部用 `../` 一層相對路徑引用 font/pic
- [ ] 餐點圖片用 `https://` 完整網址
- [ ] **已 git push 重新部署**（資源才會上線）
- [ ] 直接開 `/signage-assets/css/你的版型.css` 驗證 200
- [ ] 後台上傳菜單 → 預覽顯示完整樣式
- [ ] 建立播放清單 + 排程 → 播放器正常播放

## 五、命名與限制規則（務必遵守）

### 1. 資料夾名稱只能是 css / js / pic / font

系統的 proxy 只會改寫指向這四種資料夾的路徑。
背後邏輯在 [src/app/api/signage/asset/[id]/route.ts](../src/app/api/signage/asset/[id]/route.ts)：

```ts
html.replace(/(?:\.\.\/)+(css|js|pic|font)\//g, '/signage-assets/$1/')
```

➡️ 如果你的新版型想用別的資料夾名稱（例如 `img/`、`assets/`、`fonts/`），**菜單會抓不到資源**。
解法二選一：
- **（推薦）** 沿用 css/js/pic/font 四個資料夾
- 或請工程師在上面那行正則加入新的資料夾名稱

### 2. 檔名要獨一無二（共用資源池的代價）

目前所有版型的資源混放在同一個資料夾。所以：
- ✅ `vis_ad_machine.css`、`modern_2026.css` → 不同名，OK
- ❌ 兩個版型都叫 `pic/bg.png` 但內容不同 → **會互相覆蓋打架**

避免方式：新版型的專屬圖片**加前綴**，例如 `pic/modern_bg.png`、`pic/modern_logo.png`。

## 六、常見問題排查

| 症狀 | 原因 | 解法 |
|---|---|---|
| 菜單預覽是**裸 HTML 無樣式** | CSS 路徑錯，或資源沒部署 | 檢查 `../../css/` 寫法；確認已 git push |
| 菜單變成**下載檔案** | （此問題系統已自動處理） | proxy 已加 inline header，理論上不會發生 |
| **字型沒生效** | font 檔缺失或路徑錯 | 確認 `.ttf` 已放進 `public/signage-assets/font/` 並部署 |
| 圖片**破圖** | 圖檔沒上傳到 pic/，或檔名拼錯 | 確認檔案存在於 `public/signage-assets/pic/` |
| 改了 CSS **沒變化** | public/ 資源需重新部署 | git push 重新部署；瀏覽器 `Ctrl+Shift+R` 清快取 |
| 餐點照片**載不出** | 外部網址失效 | 檢查 `bythewaylife.com.tw` 的圖片網址是否正確 |

## 七、未來升級方向（目前用不到）

目前是**共用資源池**架構（所有版型混放、靠檔名區分），優點是系統零改動。

當版型數量變多、或經常遇到「同名檔案打架」時，可升級為**每版型獨立資料夾**：

```
public/signage-assets/templates/
├── vis_ad_machine/{css,js,pic,font}/
├── modern_2026/{css,js,pic,font}/
```

並在 `signage_assets` 資料表加一個 `template` 欄位、上傳時選版型、proxy 依版型改寫路徑。
這是平順的增量升級，需要時再請工程師處理即可。
